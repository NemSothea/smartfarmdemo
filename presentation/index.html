02 / 10 The Problem

Most developers prompt like they're texting a friend

The AI has no memory of your project, your constraints, or your past decisions.

❌ Bad prompt
"make a finance screen in SwiftUI"
✅ Good prompt
Read .claude/skills/ios.md.
Build: Views/Finance/FinanceListView.swift
Data:  FarmViewModel (ViewModels/FarmViewModel.swift)
Shows: transaction list, balance card, filter chips, bar chart
Acts:  FAB → AddTransactionSheet, swipe-to-delete
Constraint: iOS 14 — GeometryReader rects, NOT Swift Charts.
Output: complete file, no TODOs, no placeholder comments.
The difference is not cleverness — it is context + constraints + scope. The bad prompt forces the AI to guess all three.
03 / 10 Folder Structure

The 3-Layer Context System

Write the rules once. Reference them in every prompt. Never repeat yourself.

Mobile/
├── PLAN.md          ← what exists, what's done, what's next
├── FIGMA_BRIEF.md   ← design spec (iOS HIG + Material 3)
└── .claude/
    ├── memory.md    ← project rules, data models, team
    └── skills/
        ├── ios.md   ← forbidden APIs, patterns, file paths
        └── android.md ← Room, Hilt, Compose patterns
Claude Code reads these files at session start. Your constraints are loaded automatically — no copy-paste, no repeated explanations, no drift between sessions.

Layer 1 — memory.md

Project identity: data models, shared rules, currency logic, team. Stable across all sprints.

Layer 2 — skills/*.md

Platform rules: what APIs to use/avoid, architecture patterns, exact file paths. Grows with each bug you fix.

Layer 3 — PLAN.md

Sprint state: feature matrix, what is done, what is next. Updated at the end of every session.

04 / 10 Skills File

The Skills File is Your Rule Engine

Forbidden APIs enforce correctness better than any natural-language instruction.

# skills/ios.md — real excerpt

## NEVER USE
- NavigationStack       ← iOS 16+ only
- .searchable()         ← iOS 15+ only
- Charts / Chart {}    ← iOS 16+ only
- #Preview {}          ← Xcode 15+ only
- @Observable           ← iOS 17+ only
- ShareLink             ← iOS 16+ only
- if let x { }         ← Swift 5.7+ shorthand

## ALWAYS USE
- NavigationView + NavigationLink
- GeometryReader + Rectangle  ← for bar charts
- struct X_Previews: PreviewProvider
- if let x = x { }    ← explicit binding
- UIActivityViewController  ← wrapped in UIViewControllerRepresentable
Without this file, the AI writes modern Swift.
Your iOS 14 deployment target fails to build.

With this file, every generated file is deployment-target correct — automatically, every time.

Why a separate file per platform?

iOS and Android idioms are fundamentally different. Mixing them in one file creates ambiguity. One skill file = one platform = zero confusion.

When do you update it?

Every time a bug traces back to a wrong API choice → add it to NEVER USE. The file gets smarter with every sprint.

06 / 10 PLAN.md

PLAN.md is Your Memory Across Sessions

The AI forgets everything when the session ends. PLAN.md doesn't.

1
Session start: "Read PLAN.md. Sprint 5 is done. What is Sprint 6?" — AI picks up exactly where you left off, no summary needed.
2
Build iOS: "Build the iOS part of Sprint 6. Follow skills/ios.md." — Full context already loaded from step 1.
3
Build Android (same session): "Now build the Android equivalent. Follow skills/android.md." — Shared data models and feature logic are already in context.
4
Session end: "Update PLAN.md — mark Sprint 6 complete. Log any bugs found." — Next session starts from this saved state.
Why build both platforms in one session? The shared context (data models, UX rules, feature logic) is already warm. Starting a new session costs you 2–3 prompts just to re-establish context.
07 / 10 Code Quality

4 Instructions That Change Output Quality

Add these to your skills file. They apply to every prompt, forever.

1 — Architecture enforcement

"MVVM — all state in ViewModel, zero business logic in Views.
FarmViewModel owns all CoreData calls. Views only read @Published properties."

2 — No placeholders

"Complete implementation. No // TODO, no // placeholder,
no fatalError('implement me'), no incomplete switch cases."

3 — Naming consistency

"Match existing naming: FarmActivity not Activity,
FinanceEntry not Transaction, CalendarTabView not CalendarView."

4 — Safe code

"No force unwrap (!), no try! outside fatalError,
no as! unless type is compiler-guaranteed."
08 / 10 Real Bugs

Real Bugs. Real Fixes. Every bug → one new rule.

Each entry below became a permanent rule in the skills file after it happened once.

BugRoot causeRule added to skills file
NavigationStack used → build fails AI defaults to latest iOS APIs NEVER USE: NavigationStack (iOS 16+)
Bar chart bars all same height ZStack + .offset() math cancels out Use ZStack(alignment: .topLeading) + .position(x:y:) for bars
Only last .sheet fires iOS 14 honors only the final .sheet modifier Use single .sheet(item:) with Identifiable enum
Android FAB hidden behind nav bar enableEdgeToEdge + Scaffold padding conflict Always pass innerPadding to Scaffold content lambda
Dashboard shows stale data SwiftUI skips re-render when NSManagedObject pointer unchanged Extract primitive values in ForEach — never pass NSManagedObject to sub-views
The skills file is a living document. The more you build, the stronger your rule engine becomes. By Sprint 10, the AI made zero of these mistakes.
09 / 10 Sprint Pattern

The Complete Session Loop

One session = one sprint = both platforms. Repeat 10 times to ship.

── SESSION START ──────────────────────────────────────────────────────

LOAD CONTEXT
"Read PLAN.md, .claude/memory.md, .claude/skills/ios.md, .claude/skills/android.md"
→ AI now has: project state + platform rules + architecture + data models

── BUILD ───────────────────────────────────────────────────────────────

iOS  "Build [feature] for Sprint N. Follow skills/ios.md. Complete file. No TODOs."
     → Review, test in simulator

Android  "Now build the Android equivalent. Follow skills/android.md."
          → Context (data models, feature logic) already warm — no re-explanation needed

── BUG FOUND ───────────────────────────────────────────────────────────

FIX + ENCODE  "Fix [bug]. Then add the root cause as a rule to skills/ios.md."
               → Bug can never happen again

── SESSION END ─────────────────────────────────────────────────────────

SAVE STATE  "Update PLAN.md — mark Sprint N complete. Log any open issues."
             → Next session starts from this exact state
10 / 10 Summary

The 5 Golden Rules

📁
Write context once in files — reference it, never repeat it. memory.md + skills/*.md + PLAN.md are your persistent brain.
🚫
Forbidden APIs > clever words. An explicit NEVER USE list enforces constraints better than any natural-language instruction.
🎯
Name exact files and ViewModels. Never say "the finance screen." Say Views/Finance/FinanceListView.swift + FarmViewModel.
🧠
PLAN.md is your cross-session memory. Update it at the end of every sprint — next session starts cold, PLAN.md doesn't.
🔁
Every bug found = one new rule in skills/*.md. The system gets smarter with every sprint. By Sprint 10, the same mistake never happens twice.
Prompting is engineering. Treat it like writing a test spec — not a chat message. The quality of your output is determined by the quality of your context files.
